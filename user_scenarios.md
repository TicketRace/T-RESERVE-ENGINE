## UC-1: Регистрация и авторизация

**Актор**: Новый пользователь
**Предусловия**: Нет аккаунта

| Шаг | Действие | Ответ системы |
|---|---|---|
| 1 | Открывает главную страницу | Список мероприятий (недоступен без авторизации) |
| 2 | Нажимает "Регистрация" | Форма: email, пароль, имя |
| 3 | Заполняет и отправляет | POST /api/auth/register → JWT токен |
| 4 | Автоматически авторизован | Перенаправление на главную, navbar показывает имя |

**Альтернативный поток**:
- 3а. Email уже занят → ошибка "Пользователь с таким email уже существует" (409)
- 3б. Невалидный email → ошибка валидации (400)

## ⚡ UC-3: Просмотр карты мест

**Актор**: Авторизованный пользователь
**Предусловия**: Выбрал мероприятие

| Шаг | Действие | Ответ системы |
|---|---|---|
| 1 | Нажимает "Выбрать место" | GET /api/events/{id}/seats → массив мест с статусами |
| 2 | Видит карту зала (сетка рядов × мест) | Места раскрашены: 🟢 свободно, 🔴 занято |
| 3 | Фронт опрашивает сервер (polling) | GET /api/events/{id}/seats каждые 3 секунды |
| 4 | **Другой пользователь** бронирует место B-5 | Следующий polling → место B-5 стало 🔴 (задержка 0–3 сек) |
| 5 | Место B-5 освобождается (таймер истёк) | Следующий polling → место B-5 стало 🟢 |

> **Для демо**: открыть два браузера рядом, в одном кликнуть место — в другом оно меняет цвет через пару секунд (polling). MVP: polling, v2: WebSocket.

## ⚡ UC-4: Бронирование места (КРИТИЧЕСКИЙ СЦЕНАРИЙ)

**Актор**: Авторизованный пользователь
**Предусловия**: На карте мест, видит свободные места

| Шаг | Действие | Ответ системы |
|---|---|---|
| 1 | Кликает на зелёное место A-12 | Мгновенно: POST /api/bookings/lock {eventId, seatId} |
| 2 | — | **Redis SETNX** → атомарный lock на 10 минут |
| 3 | — | **PostgreSQL** UPDATE status='LOCKED' |
| 4 | — | **RabbitMQ** → delayed message (10 min) |
| 5 | — | **WebSocket** → broadcast всем: A-12 теперь LOCKED |
| 6 | Место A-12 стало 🟡 жёлтым ("ваша бронь") | Показывает: цена 1500₽, таймер ⏰ 10:00, кнопка "Оплатить" |

**Альтернативные потоки**:
- 1а. **1000 человек кликают одновременно** → только ОДИН получает 200, остальные 999 → ошибка 409 "Место уже занято"
- 1б. Место уже занято → 409 Conflict, ничего не происходит

## ⚡ UC-5: Подтверждение оплаты (бесплатно в MVP)

**Актор**: Пользователь с активной бронью
**Предусловия**: Место заблокировано (LOCKED), таймер тикает

| Шаг | Действие | Ответ системы |
|---|---|---|
| 1 | Нажимает "Оплатить" | Открывается форма выбора метода оплаты |
| 2 | Выбирает метод "Оплатить бесплатно" (он единственный) | — |
| 3 | Нажимает "Подтвердить" | POST /api/bookings/{lockId}/confirm |
| 4 | — | Backend: status LOCKED → BOOKED |
| 5 | — | WebSocket broadcast: A-12 = BOOKED |
| 6 | — | Fire-and-forget → Python сервис генерирует PDF |
| 7 | Видит "Бронирование подтверждено!" | Кнопка "Скачать билет PDF" |

**Альтернативный поток**:
- 1а. **Таймер истёк** пока юзер думал → при нажатии "Оплатить" → 410 Gone "Бронь истекла"

## ⚡ UC-6: Автоматическая отмена бронирования (Cancel Worker)

**Актор**: Система (автоматически)
**Предусловия**: Место заблокировано 10 минут назад, оплата НЕ прошла

| Шаг | Действие | Ответ системы |
|---|---|---|
| 1 | Проходит 10 минут | RabbitMQ доставляет delayed message |
| 2 | Cancel Worker получает сообщение | Проверяет status в PostgreSQL |
| 3 | Status = 'LOCKED' (не оплатили) | UPDATE status='AVAILABLE', user_id=NULL |
| 4 | — | Redis: DEL ключ блокировки |
| 5 | — | WebSocket broadcast: A-12 = AVAILABLE |
| 6 | Место снова зелёное на карте у всех | Другие пользователи могут бронировать |

**Альтернативный поток**:
- 2а. Status = 'BOOKED' (уже оплатили) → Cancel Worker ничего не делает ✅

> [!TIP]
> **Для демо**: установить таймер на 30 секунд вместо 10 минут, чтобы показать на защите в реалтайме.

## UC-8: Нагрузочное тестирование (Race Condition)

**Актор**: QA через Apache JMeter
**Предусловия**: Ивент создан, все места AVAILABLE

| Шаг | Действие | Ожидаемый результат |
|---|---|---|
| 1 | JMeter: 1000 потоков одновременно | POST /api/bookings/lock {eventId:1, seatId:"A-1"} |
| 2 | Все 1000 запросов прилетают в течение 1 секунды | — |
| 3 | Redis SETNX обрабатывает | **Ровно 1** запрос получает 200 OK |
| 4 | — | **999** запросов получают 409 Conflict |
| 5 | В PostgreSQL: место A-1 status='LOCKED' | user_id = победитель |
| 6 | В Redis: ключ `ticket:1:A-1` существует | TTL = 600 секунд |

**Метрики для отчёта**:
- Throughput: ~1000 RPS
- Avg Response Time: < 50ms
- Error Rate: 99.9% (это ПРАВИЛЬНО — 999 из 1000 должны быть ошибкой)
- Success: ровно 1

## UC-10: Генерация PDF-билета (Notification Service)

**Актор**: Система (автоматически после успешной оплаты)
**Предусловия**: Билет переведён в статус BOOKED

| Шаг | Действие | Ответ системы |
|---|---|---|
| 1 | BookingService подтверждает оплату | status = BOOKED |
| 2 | Java отправляет REST-запрос в Python сервис | POST /generate-ticket {user, event, seat, price} |
| 3 | Python (FastAPI + ReportLab) генерирует PDF | PDF содержит: название, дата, место, QR-код, лого |
| 4 | PDF сохраняется / возвращается URL | — |
| 5 | Пользователь может скачать PDF из личного кабинета | — |

**Альтернативный поток**:
- 2а. Python сервис недоступен → fire-and-forget, ошибка логируется, билет всё равно BOOKED. PDF можно сгенерировать позже повторным запросом.

> [!NOTE]
> Вызов Python сервиса — асинхронный. Если PDF не сгенерировался, это не блокирует покупку.
