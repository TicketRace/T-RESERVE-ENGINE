# 📖 Гайд по ведению разработки

Команда, привет! Чтобы мы не закопались в коде и не завалили защиту, работаем строго по правилам.

## 🌿 Git Flow (Как мы работаем с кодом)
1. **Никаких пушей в `main`!** Основная ветка всегда должна быть в рабочем состоянии (можно запустить и показать ментору).
2. Создаем ветку под свою задачу от `main`:
   - `feat/booking-logic` — новые фичи.
   - `fix/websocket-auth` — исправление багов.
   - `docs/api-specs` — документация.
3. Делаем **Pull Request (PR)**.
4. Свободный участник делает Code Review.
5. После аппрува — мерджим.

## 📝 Стандарты коммитов
Пишем кратко и понятно:
- `feat:` добавил генерацию PDF.
- `fix:` исправил ошибку в тексте.
- `docs:` обновил README.

## 🐞 Как сообщать о проблемах (Issues)
Если что-то не работает или есть идея — идем во вкладку **Issues**:
1. Жмем **New Issue**.
2. Если баг: пишем "Что ожидал" и "Что получил".
3. Если идея: пишем "Зачем это нужно".
4. Вешаем лейбл (bug, enhancement).

## 🖥 Как быстро править тексты
Если видишь ошибку в `.md` файле или сценарии:
1. Открой файл в браузере на GitHub.
2. Нажми значок **карандаша**.
3. Исправь текст.
4. Снизу выбери "Create a new branch..." и жми **Propose changes**.
5. Создастся ПР, который нужно будет замерджить.
