---

## 1. Назначение документа

Этот документ описывает общий подход к тестированию проекта.

Он не привязан к конкретным классам, методам, файлам или текущей реализации. Его задача - зафиксировать, какие виды тестов нужны проекту, как оценивать покрытие и когда тестирование можно считать достаточным.

Главный принцип: тесты должны подтверждать не только то, что код выполняется, но и то, что система сохраняет корректное состояние при ошибках, параллельных запросах и пограничных сценариях.

---

## 2. Основные цели тестирования

Тестирование в проекте должно подтверждать, что:

- пользовательские сценарии работают предсказуемо;
- одно и то же место нельзя забронировать дважды;
- ошибки обрабатываются контролируемо;
- состояние данных не повреждается при неуспешных операциях;
- защищённые действия недоступны без нужных прав;
- API остаётся понятным и стабильным для фронта и ручного тестирования;
- критичные сценарии можно проверить автоматически перед merge или release.

---

## 3. Используемые тестовые подходы

### 3.1 Risk-based тестирование

В первую очередь покрываются зоны с самым высоким риском:

| Зона | Почему критична |
|---|---|
| Конкурентное бронирование | Ошибка может привести к двойной продаже места |
| Переходы состояний | Неверный статус ломает дальнейший пользовательский flow |
| Авторизация и роли | Ошибка может открыть доступ к чужим или административным действиям |
| Работа с БД | Ошибка может повредить данные или нарушить ограничения |
| API-контракт | Фронт и ручное тестирование зависят от стабильного поведения API |

Менее рискованные сценарии покрываются после критических.

---

### 3.2 Пирамида тестирования

В проекте используется пирамида тестирования:

| Уровень | Назначение | Особенность |
|---|---|---|
| Unit | Быстро проверить отдельную бизнес-логику | Без реальной инфраструктуры |
| Integration | Проверить работу нескольких слоёв вместе | С реальной БД и миграциями |
| API | Проверить поведение системы через HTTP | Близко к поведению пользователя или фронта |
| Load | Проверить устойчивость под нагрузкой | Используется для ключевых NFR |

Основной объём должен быть на unit и integration уровнях. API и нагрузочные тесты должны покрывать только важные пользовательские и технические сценарии.

---

### 3.3 Shift-left тестирование

Тесты должны появляться вместе с изменениями, а не после завершения разработки.

Для каждой задачи желательно сразу определить:

- какие позитивные сценарии должны быть проверены;
- какие ошибки возможны;
- какие состояния данных должны измениться;
- какие состояния данных не должны измениться;
- нужны ли интеграционные или concurrency-тесты.

---

### 3.4 Scenario-based тестирование

Для проекта важнее не просто процент покрытия, а покрытие сценариев.

Тест должен проверять законченное поведение:

```text
условие -> действие -> ожидаемый результат -> итоговое состояние системы
```

Например, недостаточно проверить только код ответа. Нужно также проверить, что данные после операции находятся в правильном состоянии.

---

### 3.5 Конкурентное тестирование

Для системы бронирования обязательны тесты конкурентного доступа.

Минимальный обязательный сценарий:

```text
несколько пользователей одновременно пытаются занять один и тот же ресурс
```

Ожидаемый результат:

- ровно одна операция завершается успешно;
- остальные получают контролируемый конфликт;
- в базе нет дублей;
- итоговое состояние ресурса однозначное;
- нет пятисоток.

Такие тесты должны выполняться на реальной БД. Моки не подходят для доказательства корректной работы блокировок, транзакций и ограничений базы данных.

---

### 3.6 API тестирование

API-тесты нужны для проверки поведения системы снаружи.

Они должны покрывать:

- успешные пользовательские сценарии;
- ошибки валидации;
- отсутствие авторизации;
- недостаточные права;
- повторные или конфликтующие действия;
- соответствие фактического поведения API-документации.

Postman-коллекция используется как ручной и демонстрационный набор. Она не заменяет автоматические unit, integration и concurrency-тесты.

---

### 3.7 Регрессионное и smoke тестирование

Smoke-тесты проверяют, что система в целом стартует и базовые сценарии работают.

Regression-тесты защищают уже реализованное поведение от поломок при новых изменениях.

Минимальный smoke-набор должен проверять:

- старт приложения;
- применение миграций;
- доступность health-check;
- базовый пользовательский flow;
- базовые проверки авторизации.

---

## 4. Методика оценки покрытия

Покрытие оценивается не одним числом, а несколькими показателями.

| Вид покрытия | Что показывает |
|---|---|
| Line coverage | Какая часть кода была выполнена тестами |
| Branch coverage | Проверены ли разные ветки условий и ошибок |
| Scenario coverage | Покрыты ли важные бизнес-сценарии |
| State coverage | Проверяется ли итоговое состояние данных |
| Contract coverage | Проверено ли поведение публичного API |
| Risk coverage | Закрыты ли самые опасные зоны продукта |

Процент покрытия полезен как индикатор, но он не должен быть единственным критерием качества.

---

## 5. Критерии достаточности покрытия

### 5.1 Unit-тесты считаются достаточными, если

- покрыты основные позитивные сценарии;
- покрыты основные негативные сценарии;
- проверяются ошибки и исключительные ситуации;
- проверяются допустимые и недопустимые переходы состояний;
- проверяются побочные эффекты успешных операций;
- проверяется отсутствие побочных эффектов при ошибках;
- тесты быстрые, стабильные и не требуют внешней инфраструктуры.

---

### 5.2 Интеграционные тесты считаются достаточными, если

- используются реальные миграции БД;
- тесты выполняются на той же СУБД, что и проект;
- проверяются ограничения, индексы и уникальность данных;
- проверяются транзакции и блокировки там, где они важны;
- после операции проверяется состояние данных в БД;
- тесты можно запустить и локально, и в CI.

---

### 5.3 Concurrency-тесты считаются достаточными, если

- несколько операций стартуют одновременно;
- тест использует реальную БД;
- есть один ожидаемый успешный результат;
- конфликтующие операции завершаются контролируемой ошибкой;
- после завершения всех операций проверяется состояние данных;
- результат стабилен при повторных запусках;
- в результате нет технических ошибок уровня 5xx.

---

### 5.4 API-тесты считаются достаточными, если

- каждый публичный сценарий имеет хотя бы одну успешную проверку;
- для каждого важного сценария есть негативная проверка;
- защищённые действия проверяются без токена и с недостаточными правами;
- проверяется формат ответа и ключевые поля;
- проверяется корректность бизнес-ошибок;
- Postman-коллекция может использоваться для демо и ручной проверки.

---

### 5.5 Нагрузочные тесты считаются достаточными, если

- выбран конкретный критичный сценарий;
- задана ожидаемая нагрузка;
- проверяются не только успешные ответы, но и допустимые бизнес-конфликты;
- технические ошибки отделяются от ожидаемых бизнес-ошибок;
- фиксируются response time, error rate и количество успешных/конфликтных операций;
- результат можно воспроизвести повторно.

---

## 6. Минимальные ориентиры покрытия

| Область | Минимальное ожидание |
|---|---|
| Критичная бизнес-логика | 80%+ line coverage и покрытие основных веток |
| Переходы состояний | Все допустимые и недопустимые переходы проверены сценарно |
| Конкурентный доступ | Обязательный integration/concurrency-тест на реальной БД |
| Публичный API | Минимум positive и negative checks для ключевых сценариев |
| Авторизация | Проверка доступа без токена, с неверным токеном и с недостаточными правами |
| Ошибки | Бизнес-ошибки не должны превращаться в 5xx |

Если инструмент измерения покрытия ещё не подключён, достаточность временно оценивается по чек-листу сценариев.

---

## 7. Критерии полноты для тестов

Задача считается достаточно покрытой тестами, если:

- добавлены тесты на успешные сценарии;
- добавлены тесты на основные ошибки;
- проверяется итоговое состояние данных;
- проверяется отсутствие нежелательных изменений при ошибках;
- для конкурентных сценариев есть проверка на реальной БД;
- API-поведение можно проверить автоматически или через Postman;
- все тесты проходят локально;
- в PR указано, какие проверки были добавлены и как их запускать.

---

## 8. Что смотреть на PR review

При проверке тестов важно смотреть не только на количество тестов.

Хороший тест должен:

- иметь понятное название;
- проверять один сценарий;
- явно фиксировать ожидаемый результат;
- проверять состояние данных после операции;
- проверять ошибки отдельно от успешных сценариев;
- не зависеть от случайных задержек;
- не мокать инфраструктуру там, где важно проверить её реальное поведение;
- быть стабильным при повторном запуске.

Плохой признак - тест проходит, но не доказывает важное бизнес-правило.

---

## 9. Короткое правило

Для обычной логики достаточно unit-тестов.  
Для работы с БД нужны integration-тесты.  
Для конкурентного бронирования нужен concurrency-тест на реальной БД.  
Для публичного API нужны API/Postman-проверки.  
Для релиза нужны smoke и regression проверки.
