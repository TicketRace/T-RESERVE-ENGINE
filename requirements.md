# T-RESERVE ENGINE — Требования

> Версия 1.0 | Дата: 14.04.2026

---

## Функциональные требования

| # | Требование | Связанный UC |
|---|---|---|
| FR-1 | Возможность регистрации пользователя с указанием электронной почты, имени и пароля | UC-1 |
| FR-2 | Возможность авторизации зарегистрированного пользователя по электронной почте и паролю | UC-1 |
| FR-3 | Возможность просмотра списка мероприятий после авторизации | UC-2 |
| FR-4 | Возможность просмотра карты мест выбранного мероприятия после авторизации | UC-3 |
| FR-5 | Возможность бронирования места авторизованным пользователем | UC-4 |
| FR-6 | Возможность оплаты забронированного билета авторизованным пользователем | UC-5 |
| FR-7 | Возможность просмотра личного кабинета с купленными билетами | UC-9 |
| FR-8 | Возможность автоматической отмены бронирования системой, если пользователь не оплатил билет в течение 10 минут | UC-6 |
| FR-9 | Возможность создания нового мероприятия администратором с указанием названия, описания, иллюстрации, даты, длительности, площадки, цены и возрастного ограничения | UC-7 |
| FR-10 | Возможность проведения нагрузочного тестирования системы бронирования с помощью JMeter | UC-8 |

---

## Нефункциональные требования

| # | Требование | Категория |
|---|---|---|
| NFR-1 | Система должна обрабатывать более 1000 запросов в секунду на бронирование мест. Блокировка места должна происходить быстрее 50 миллисекунд | Производительность |
| NFR-2 | При одновременном бронировании одного места тысячью пользователями (тысяча человек нажали на кнопку в течение 1 секунды) только один должен получить успех, остальные должны получать сообщение, что место уже занято | Консистентность |
| NFR-3 | Серверная часть не должна хранить состояние пользователя, чтобы можно было запускать несколько копий приложения | Масштабируемость |
| NFR-4 | Карта мест должна обновляться в реальном времени через WebSocket без перезагрузки страницы | UX / Realtime |
| NFR-5 | Система должна успешно проходить тест на 1000 одновременных бронирований одного места с помощью JMeter | Тестируемость |
| NFR-6 | Время отклика интерфейса при базовых операциях (создание/редактирование) не должно превышать 1.5 секунды при стандартной нагрузке (50-100 пользователей в минуту) | Производительность |
| NFR-7 | Интерфейс должен быть разработан с использованием адаптивной верстки, обеспечивающей корректное отображение на всех разрешениях экрана от 320px (мобильные устройства) до 1920px (десктоп) | Адаптивность |
| NFR-8 | На стороне клиента и сервера должна быть реализована валидация вводимых данных для предотвращения некорректных запросов и сбоев | Надёжность |
| NFR-9 | Процесс регистрации должен занимать не более 1 минуты и требовать заполнения только трёх полей (email, имя, пароль) | Удобство |
| NFR-10 | Система должна функционировать в рамках современных версий браузеров (Chrome, Firefox, Safari, Edge) | Совместимость |

## Как требования реализованы в архитектуре

| Требование | Техническое решение |
|---|---|
| NFR-1 (1000 RPS, <50ms) | Redis SETNX — атомарный lock за O(1) |
| NFR-2 (Race Condition) | SETNX однопоточный → первый получает lock, остальные NIL → 409 |
| NFR-3 (Stateless) | JWT-токен содержит userId и role. Сервер не хранит сессии |
| NFR-4 (Realtime) | WebSocket STOMP broadcast при каждом изменении статуса места |
| NFR-5 (JMeter тест) | Сценарий UC-8: 1000 потоков → 1 success, 999 conflict |
| NFR-6 (<1.5s UI) | React SPA + кэширование |
| NFR-7 (Адаптивность) | CSS media queries, 320px–1920px breakpoints |
| NFR-8 (Валидация) | Server: `@Valid` + Bean Validation (JSR-380). Client: React form validation |
| NFR-9 (Регистрация <1 мин) | Три поля: email, имя, пароль. Без email verification |
| NFR-10 (Браузеры) | React + Vite: поддержка всех modern browsers из коробки |
